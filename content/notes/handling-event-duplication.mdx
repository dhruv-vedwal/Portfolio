In distributed architectures, message brokers (like RabbitMQ, Apache Kafka, or AWS SQS) guarantee **At-Least-Once Delivery**. This means that while messages are guaranteed to reach their destination, network drops, server restarts, or delayed acknowledgments will eventually cause the same message to be delivered **multiple times**.

If your backend consumer increments account balances, captures credit cards, or updates inventory, processing a duplicate event will corrupt your data.

Here is how to design an **Idempotent Event Consumer** to guarantee zero data duplication under all network circumstances.

---

### The Anatomy of a Duplicate Event

Consider the following scenario:

```txt
┌───────────┐         ┌────────────────┐         ┌────────────┐
│ Publisher │ ──────► │ Message Broker │ ──────► │  Consumer  │
└───────────┘         └────────────────┘         └────────────┘
                            │                          │
                            │ ─── (Delivers Event) ──► │  (Processes successfully)
                            │                          │
                            │ ◄── (Ack delayed/lost) ─  │  (Server crashes mid-ack)
                            │                          │
                      (Redelivers Event)
                            │                          │
                            │ ─── (Delivers Event) ──► │  (PROCESSED TWICE!)
```

Because the acknowledgment was lost or delayed, the broker assumes the consumer failed and redelivers the message, leading to a duplicate write.

---

### Step 1: Idempotency Keys (The Contract)

To prevent duplication, every event must carry a unique identifier—an **Idempotency Key**. This key should be generated by the producing service at the moment the event occurs (such as a UUID representing the event):

```json
{
  "event_id": "evt_8f3d1b9a-4c2d-9e1b-7a3f5c2d9a1b",
  "type": "payment.succeeded",
  "amount": 2999,
  "currency": "usd"
}
```

---

### Step 2: The Redis Deduplication Filter

Before executing heavy database queries or charge operations, the consumer checks the incoming `event_id` against a high-speed Redis distributed cache:

```typescript
// Redis-based idempotency check
async function processEvent(event: AppEvent) {
  const cacheKey = `idempotency:event:${event.event_id}`;
  
  // Try to set the key with a 24-hour expiration (TTL)
  // NX = Only set if it does not already exist
  const acquired = await redis.set(cacheKey, "processing", "EX", 86400, "NX");
  
  if (!acquired) {
    console.warn(`Duplicate event discarded: ${event.event_id}`);
    return; // Acknowledge and discard safely
  }

  try {
    // Execute business transactions...
    await executeBillingTransaction(event);
    
    // Mark as completed
    await redis.set(cacheKey, "completed", "EX", 86400);
  } catch (error) {
    // Clean up Redis key on execution failure to allow retries
    await redis.del(cacheKey);
    throw error; // Re-throw to trigger queue retry
  }
}
```

#### Why Redis is ideal:
* **Speed**: Operations execute in under a millisecond, protecting your backend from processing delays.
* **Atomic Flags**: Redis `SET NX` operations are atomic, guaranteeing that even if a duplicate event is processed concurrently on two separate workers, only one will succeed in acquiring the processing flag.

---

### Step 3: Relational Database Constraints (The Safety Net)

While Redis is exceptionally fast, a server crash or cache eviction can leave your backend exposed. For ultimate safety, establish a unique constraint directly in your relational database:

```sql
ALTER TABLE billing_transactions ADD CONSTRAINT unique_event_id UNIQUE (event_id);
```

If a duplicate event slips past the Redis check, the database transaction will trigger a unique constraint violation error, aborting the write and rolling back changes, keeping your data clean.

---

### Key Architectural Guidelines

1. **At-Least-Once vs. Exactly-Once**: Truly exactly-once delivery is a mathematical impossibility across decoupled networks. Design your consumers to assume duplicate deliveries will happen, focusing on idempotent patterns.
2. **Proper Expiration (TTL)**: Keep your idempotency keys in Redis for at least **24 to 72 hours**. Most retry bursts occur within a few hours of network outages.
3. **Safe Return for Duplicates**: When discarding duplicates, always return a success status to the broker. If you return an error, the broker will continue redelivering the event indefinitely, causing queue clog.

### Conclusion

Handling event duplication is mandatory to build reliable, event-driven software. By establishing unique idempotency keys, implementing an atomic Redis filter, and reinforcing unique constraints in your primary database, you build a resilient pipeline that handles erratic network failures and retries seamlessly without compromising data integrity.
